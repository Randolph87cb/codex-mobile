# 2026-05-19 Android 应用日志能力

## 目标

- 为 Android 端补上可持续保留、可在应用内查看的日志能力，方便后续排查 bridge 连接、会话实时流和发送输入相关问题。

## 当前方案

- 新增轻量级本地日志器，写入应用私有目录。
- 在 bridge 数据层和 `AppViewModel` 关键路径接入日志。
- 在设置页提供最近日志查看、刷新、清空和复制入口。

## 执行中

- 已完成 Android 数据层与设置页梳理，按轻量方案落地。

## 关键改动

- 新增 `android/app/src/main/java/com/openai/codexmobile/diagnostics/AppLogger.kt`，提供本地文件日志、Logcat 镜像、日志读取、清空和未捕获异常记录。
- `MainActivity.kt` 与 `ReplayHarnessActivity.kt` 接入日志器，并把它注入 `AppViewModelFactory`。
- `RealBridgeDataProvider.kt` 记录 bridge HTTP/WS 关键链路，包括连接、会话创建、输入发送、审批、实时流事件与失败信息。
- `AppViewModel.kt` 记录连接、打开会话、草稿转正式、排队发送、审批与实时流状态变化，并把最近日志暴露给 UI。
- `SettingsScreen.kt` 新增“应用日志”卡片，支持刷新、清空、复制日志；`CodexMobileApp.kt` 接上剪贴板操作。
- `AppViewModelTest.kt` 补了日志状态的单元测试，并更新现有测试构造。

- 已补一条设置页日志交互的 `androidTest` 回放用例，验证日志卡片可见、复制日志、清空日志和刷新日志这条链路在模拟器上可跑通。

## 线上排查补充

- 用户在真机上反馈同一线程发送消息后报错；从新增日志中定位到根因是 bridge 把会话配置中的 `serviceTier=flex` 继续传给了底层 `codex app-server`，而对方返回了 `Unsupported service_tier: flex`。
- 进一步直接查询本机 `codex-cli 0.128.0` 的 `model/list`，确认当前运行时对 `gpt-5.4` 暴露的是 `additionalSpeedTiers=["fast"]`，没有 `flex`。因此问题不是 Android 乱传值，而是“协议层存在 flex，但当前模型/运行时未对该模型开放 flex”。
- 已撤掉 bridge 里的静默回退逻辑，改成在 `thread/start`、`thread/resume` 和 `turn/start` 前先用 `model/list` 显式校验当前模型支持的速度档位；如果当前模型不支持 `flex`，bridge 会明确报错并把会话状态置为 `error`，不再偷偷切到 `fast`。
- 已补 bridge 单元测试覆盖“`gpt-5.4` 仅支持 `fast` 时拒绝 `flex`”的场景，并执行 `cd bridge; npm run check; npm test` 验证通过。
