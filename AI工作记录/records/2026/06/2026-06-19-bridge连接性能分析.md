# 2026-06-19 bridge 连接性能分析

## 目标

排查手机连接一个 bridge 时间过长的问题，拆分 Android 连接流程、bridge HTTP 接口和 codex app-server JSON-RPC 调用的实际耗时。

## 实测结果

- `mock` runner 基线：`/health` 约 43ms，`/api/sessions` 约 3-8ms。
- `app-server` runner：`/health` 约 11-36ms，首次 `/api/sessions?archived=false` 约 681-2167ms，热请求约 481ms。
- JSON-RPC 拆分：`initialize/start` 约 185ms，`thread/list archived=false` 约 446ms，`thread/list archived=true` 约 93ms，`account/rateLimits/read` 约 1793ms。
- 复现 Android 连接路径时，25 个历史会话都被判断为需要同步托管策略，串行触发 25 次 `PATCH /api/session/:id/config`，合计约 14045ms，单次最高约 2784ms。

## 当前判断

主要瓶颈不是 `/health` 或局域网连接，而是 Android 连接成功后同步历史会话策略：历史会话摘要默认显示为 `manual/workspace-write`，Android 认为它们不满足托管策略，于是逐个调用配置更新。每次更新会触发 bridge 侧 attach/resume/read 历史线程，造成连接阶段被 N 个历史会话数量放大。

次要耗时是 bridge 的 `/api/sessions?archived=false` 在 app-server 模式下会额外查询归档列表，用于过滤本地 store 中的归档线程；额度接口约 1.5-1.9s，但它发生在 UI 更新连接成功之后。

## 待确认优化方向

- 首选：去掉 Android 连接阶段对列表中每个历史会话的自动策略同步，只在真正打开或创建会话时同步必要策略。
- bridge 可同步优化：`listSessionViews(false)` 避免无条件查询归档列表，只有本地 store 非空时才需要查归档用于过滤。
- 可选：额度读取改为后台刷新或缓存，避免用户把额度加载误感知为连接未完成。

## 实施结果

- Android 已移除连接/列表刷新阶段的批量策略同步，改为只在本地展示时归一化为托管策略；打开会话后的按需修正逻辑保留。
- bridge 已优化当前会话列表：本地 store 为空或没有未知本地会话时，不再额外查询归档线程列表。
- 新增 Android 回归测试覆盖连接时不批量调用 `updateSessionConfig`，同时保持列表和默认详情本地托管策略展示。
- 新增 bridge 回归测试覆盖 store 为空时只查当前线程列表，以及 store 有未知本地记录时仍查询归档列表过滤。

## 验证

- `cd bridge; npm run check` 通过。
- `cd bridge; npm test` 通过，77 tests passed。
- `powershell -ExecutionPolicy Bypass -File .\scripts\build-android-debug.ps1` 通过。
- `cd android; .\gradlew.bat testDebugUnitTest` 通过。
- 优化后实际计时：`/health` 约 28ms，`/api/sessions?archived=false` 约 632ms，首个详情约 15ms；连接阶段不再出现 25 次策略同步 PATCH。额度接口仍约 1583ms，后续可单独缓存或后台化。
