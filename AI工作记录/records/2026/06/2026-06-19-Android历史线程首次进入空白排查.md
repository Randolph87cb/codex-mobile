# Android 历史线程首次进入空白排查

## 背景

- 现象：手机进入已有线程时，页面可能只显示系统工作目录等元信息，长时间不补齐历史消息；返回线程列表后重新进入则正常加载。
- 截图线程：`019e96bf-c472-7e91-a1bb-d1908003ff70`。

## 排查结论

- 根因定位为首次进入历史线程时 Android 自动修正托管策略的返回值覆盖了原始详情快照。
- `AppViewModel.openSessionDetail()` 首次 `GET /api/session/:id` 后调用 `ensureManagedSessionPolicy()`。
- `ensureManagedSessionPolicy()` 在审批模式或 sandbox 未符合托管策略时调用 `PATCH /api/session/:id/config`，并把 PATCH 返回值作为详情继续显示。
- bridge 的 `updateSessionConfig()` 返回的是 `SessionRecord`，不是带完整 turns 的历史 `SessionView`；客户端解析后只得到本地元信息 transcript，例如“工作目录/线程 ID/当前轮次/最近错误”。
- 第二次进入时策略已经被修正，不再触发 PATCH 覆盖，所以直接使用完整 GET 结果，表现为返回列表后重新进入正常。

## 证据

- `.logs/bridge/bridge-stdout.log` 中同一线程首次进入序列：`GET /api/session/019e96bf...` -> `PATCH /api/session/019e96bf.../config` -> `GET /api/session/019e96bf.../ws`。
- 稍后重新进入序列：再次 `GET /api/session/019e96bf...` 后没有同样的策略修正 PATCH，页面恢复。
- 当前直接请求 `GET http://127.0.0.1:8787/api/session/019e96bf-c472-7e91-a1bb-d1908003ff70` 可返回完整 `transcriptPreview`，长度约 8783 字符。
- 相关代码：
  - `android/app/src/main/java/com/openai/codexmobile/AppViewModel.kt`：`ensureManagedSessionPolicy()` 使用 `updateSessionConfig()` 返回值。
  - `bridge/src/session-service.ts`：`updateSessionConfig()` 返回 `SessionRecord`。
  - `bridge/src/session-view.ts`：`buildSessionViewFromRecord()` 的 transcript 是本地元信息。

## 建议修复

- 首选：Android 侧在 `ensureManagedSessionPolicy()` 后保留原始 `detail.transcriptPreview`，用 `mergeSessionDetail(detail, patchedDetail)` 合并 PATCH 返回值，而不是让 PATCH 返回值直接替代详情。
- 或者 bridge 侧让 `PATCH /api/session/:id/config` 返回 `getSessionView()` 的完整视图，但这会改变接口行为，影响面更大。
- 需要补 Android 单元测试：模拟首次详情 GET 返回完整历史、随后策略 PATCH 返回本地元信息，验证最终 `selectedSession.transcriptPreview` 保留完整历史。

## 已实施修复

- `AppViewModel.ensureManagedSessionPolicy()` 在策略修正 PATCH 返回后改为 `mergeSessionDetail(detail, updated)`。
- 复用现有 `chooseMoreCompleteTranscript()`，避免 PATCH 返回的本地元信息预览覆盖首次 GET 已拿到的完整历史 transcript。
- 新增单元测试 `openingManagedSessionPolicyFixDoesNotDiscardMoreCompleteTranscript`，覆盖“首次详情完整、PATCH 返回较短元信息”的场景。
