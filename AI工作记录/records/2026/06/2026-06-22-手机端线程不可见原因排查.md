# 手机端线程不可见原因排查

## 背景

- 用户询问：为什么现在有的线程在手机端看不到。
- 前半段做只读排查；确认根因后，按用户确认的 harness + subagent 方案修复 bridge 侧历史线程回退。

## 排查要点

- Android 会话列表调用 `RealBridgeDataProvider.listSessions(archived)`，请求 bridge 的 `GET /api/sessions?archived=true|false`。
- bridge 的 `/api/sessions` 由 `sessionService.listSessionViews()` 提供；app-server 模式下转到 `AppServerRunner.listSessionViews(archived)`。
- `AppServerRunner.listSessionViews(false)` 读取 `thread/list archived=false`，并排除已归档线程，避免本地 `SessionStore` 残留把归档线程重新混回当前列表。
- Android UI 默认 `showArchivedSessions=false`，默认显示“当前线程”；切换到“归档线程”后才请求 `archived=true`。

## 当前本机证据

- `GET http://127.0.0.1:8787/health` 正常，runnerMode 为 `app-server`。
- `GET http://127.0.0.1:8787/api/sessions?archived=false` 返回 25 条当前线程。
- `GET http://127.0.0.1:8787/api/sessions?archived=true` 返回 25 条归档线程。

## 结论

- 当前最主要原因不是手机端解析丢数据，而是列表被分成“当前线程”和“归档线程”两个集合。
- 默认列表只显示未归档线程；已归档线程需要切换到“归档线程”页签查看，或恢复归档后回到当前列表。

## 追加排查：自我成长新线程缺失

- 用户指出：2026-06-21 或 2026-06-22 在 Codex App 的“自我成长”项目里创建的新线程，在手机端当前/归档列表都看不到。
- 找到候选线程：
  - id：`019eeada-1d08-7863-9851-a84bc0daedd0`
  - 文件：`C:\Users\Administrator\.codex\sessions\2026\06\21\rollout-2026-06-21T23-43-48-019eeada-1d08-7863-9851-a84bc0daedd0.jsonl`
  - `cwd`：`D:\workspace\自我成长`
  - 用户首条消息是关于“重复任务厌倦/自我消耗”的问题。
- 该 JSONL 文件结构正常，包含 `session_meta`、`turn_context`、用户消息、助手消息和 `task_complete`。
- 但通过当前 bridge 访问 `GET /api/session/019eeada-1d08-7863-9851-a84bc0daedd0` 返回 404。
- 当前 bridge 的 `GET /api/sessions?archived=false` 和 `GET /api/sessions?archived=true` 中也都没有该线程。

## 更新结论

- 这条缺失线程不是 Android UI 筛选问题，也不是归档/未归档切换问题。
- 根因在 bridge 的历史线程来源：bridge 只通过 app-server 的 `thread/list` / `thread/read` 获取历史线程；该线程虽然已经落到 `.codex\sessions` JSONL 文件和 `session_index.jsonl` 中，但当前 app-server 没有把它纳入可读线程集合，因此 bridge 返回不到，手机端也无法显示。
- 后续修复方向应考虑：
  - 让 bridge 在 app-server 404/列表缺失时，从 `session_index.jsonl` 或 `.codex\sessions` JSONL 做只读 fallback；
  - 或确认重启 bridge/app-server 是否能刷新 app-server 的历史线程索引，再决定是否需要代码 fallback。

## app-server 能力边界确认

- 协议生成物里的 `ThreadListParams` 支持 `limit`、`cursor`、`cwd`、`sourceKinds`、`searchTerm`、`useStateDbOnly`。
- 注释显示 `thread/list` 默认会扫描 JSONL rollouts 修复 metadata，`useStateDbOnly=true` 才跳过扫描。
- 直接启动一次性 app-server 客户端测试：
  - `thread/read` 对 `019eeada-1d08-7863-9851-a84bc0daedd0` 后续可以读出标题“改善重复任务厌倦”和 `cwd=D:\workspace\自我成长`。
  - 但 `thread/list` 即使使用 `limit=100/300`、`cwd=D:\workspace\自我成长`、全量 `sourceKinds`，仍不列出该线程。
  - `thread/loaded/list` 只描述内存中已加载线程，不是历史索引刷新接口。
- 当前判断：app-server 有“按已知 threadId 读 rollout”的能力，但没有在协议里发现可靠的“刷新/重建历史列表索引”接口；而手机端列表需要先发现 threadId，因此 bridge 需要自己的只读本地索引 fallback。

## 修复实现记录

- 新增 bridge 侧 `LocalHistoryStore`，只读扫描 Codex 本地历史：
  - 未归档：`%USERPROFILE%\.codex\sessions\**\rollout-*.jsonl`
  - 已归档：`%USERPROFILE%\.codex\archived_sessions\rollout-*.jsonl`
  - 线程标题优先读取 `session_index.jsonl`。
- `/api/sessions?archived=false|true` 仍先保留 app-server 返回结果，再补本地 rollout 漏项；以 `id/threadId` 去重，app-server 优先，并按 `lastUpdated` 降序。
- `/api/session/:id` 在 app-server 返回空或读取抛错时，尝试从本地 rollout 构造兼容 `SessionView` 的详情。
- 本地 JSONL 解析失败时跳过单文件，不影响列表接口。
- 本地扫描带短 TTL 缓存，避免每次请求都重复全量扫描；列表页只读 rollout 文件开头用于生成预览，详情页按 ID 命中后再读取完整文件。
- 未修改 Android；未修改真实 `.codex` 用户历史文件。

## 追加修复：隐藏内部 subagent 线程

- 用户确认按 harness + subagent 方案修复“手机端能看到 subagent 线程”的问题；subagent 负责 bridge 侧初版 patch，主线程负责复核、补强、验证、提交和 push。
- 根因：bridge 新增本地历史 fallback 后，会把 Codex 内部 subagent rollout 也当作普通历史线程补到 `/api/sessions`。
- 新增 `bridge/src/thread-visibility.ts`，只按稳定内部标记过滤：
  - `thread_source === "subagent"`
  - `threadSource === "subagent"`
  - `source.subagent` 字段存在
- `AppServerRunner` 在 `thread/list`、`thread/read`、`thread/resume` 入口过滤内部 subagent；直接读取 subagent 详情返回空，由路由表现为 404。
- `LocalHistoryStore` 在解析 `session_meta.payload` 后过滤内部 subagent；列表 fallback 和详情 fallback 都不返回。
- 补充测试：
  - `thread-visibility` helper 只隐藏稳定 subagent 标记，只有 `parent_thread_id` / `parentThreadId` 不隐藏。
  - app-server active/archived `thread/list` 返回 subagent 时列表不包含。
  - app-server 直接 `getSessionView(subagentId)` 返回 `null`。
  - 本地 rollout fallback 扫到 `thread_source=subagent` 时列表不包含，直接 `GET /api/session/local-subagent` 返回 404。

  - `bridge`: `npm test` 通过，8 个测试文件、88 个测试。
- Git：subagent 按要求未提交、未 push；主线程复核后提交并推送。

## 追加排查：历史线程进入详情时报 `session-not-found`

- 用户反馈：进入“实现后台收消息”线程后，详情页顶部实时流报错 `session-not-found`。
- 只读确认：
  - `GET /api/session/019eeb08-f9da-75a3-a257-77c161817a61` 可以返回完整历史详情，来源为 `history`。
  - `GET /api/session/:id/ws` 走 `getRealtimeSessionState()`，只查 app-server 可订阅状态和内存 `SessionStore`，没有走本地历史详情 fallback。
- 根因：历史-only 线程“详情可读”和“实时订阅可用”是两个能力边界；bridge WebSocket 入口把不能实时订阅的历史线程误判为线程不存在。
- 修复：
  - `/api/session/:id/ws` 在实时状态不存在时，先调用 `getSessionView()`。
  - 如果能读到历史详情，则发送一次 `session.started` 快照并保持连接，不再返回 `session-not-found`。
  - 如果详情也不存在，才保留原来的 `session-not-found` 错误。

  - `bridge`: `npm run check` 通过。
  - `bridge`: `npx vitest run tests/local-history-fallback.test.ts` 通过。
  - `bridge`: `npm test` 通过，8 个测试文件、88 个测试。
